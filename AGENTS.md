# Project state (verified 2026-07-07 — see lingrazul_construct.docx for the older, partly-superseded original plan)

This section is the source of truth for where the app actually is. The original construct plan (`lingrazul_construct.docx`) described a `content_items` / `user_progress` / `lessons` schema with a Node+Express backend — that plan is largely historical at this point; don't assume anything from it without checking here first. Re-verify this section against the live Supabase project and codebase before relying on it for anything load-bearing — it reflects a point-in-time check, not a live sync.

## Stack

No backend exists — the Expo client (`lib/supabase.ts`) talks directly to Supabase using `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY` from `.env` (gitignored). No Vercel/Railway, no Whisper, no Claude API integration. Web is deployed via **EAS Hosting** (see Deployment section below) — `expo.web.output` is set to `"static"` in `app.json`.

## Supabase schema (live)

- `letters_japanese` (id, hiragana, katakana, romaji, kanji, has_kanji, difficulty, category, tags text[]). Populated: base kana (vowel/k-row/.../w-row/n-standalone), dakuten/handakuten, youon (`insert_youon_letters.sql`), loanword-only extended katakana (`insert_katakana_extended_letters.sql`), and ~109 JLPT N5 kanji rows with one reading each (`insert_n5_kanji.sql`, category `kanji`, difficulty 4 — schema only supports one reading per row, flagged as a known simplification).
- `words_japanese` (id, text), `words_english` (id, text), `words_chinese` (id, text — unused by app code).
- `word_descriptions` (id, description, category, difficulty, tags text[] — joins to the `words_*` tables by shared id). The `tags` column was added late via `supabase/sql/add_tags_to_word_descriptions.sql` (`alter table ... add column tags text[] default '{}'`) — confirmed run against the live DB. No real tag content has been authored for word rows though (they're all empty arrays) — that's a separate content-curation pass, distinct from the schema change, and still outstanding.
- `users` (id — must equal the `auth.users` UID, email, style_preference, created_at). Auto-provisioned via a DB trigger (`users_auto_provision.sql`) plus a client-side self-healing fallback (`lib/users.ts`'s `ensureUserRow`, called from `login.tsx` after every successful sign-in). `style_preference` is for an onboarding flow that was never built — safe to leave null.
- `user_progress` (user_id, item_id, exposures, accuracy, self_report, last_seen). `item_id` intentionally has **no foreign key** (`drop_user_progress_item_fk.sql` dropped it) because it can point at either `word_descriptions` or `letters_japanese` and there's no `content_type` column to disambiguate. **Superseded as of `v2_progress_schema.sql`** — the app no longer reads or writes this table at all (`lib/progress.ts`/`lib/stats.ts` use `letter_progress`/`word_progress`/`kanji_progress` instead, see below). Left in place, untouched, not migrated from. `self_report` was already dead weight before the supersession — the "Too hard"/"Too easy" buttons that wrote to it were removed from the UI (redundant once the difficulty filter existed), and the column was dropped entirely in the v2 replacement.
- No `lesson_ratings`, `lesson_stats`, `copyright_claims`, or `sessions` tables — `community.tsx` (messaging/forums/user-authored lessons) is UI-only (see below). A `lessons`/`lesson_sections` pair DOES now exist (see `v2_lessons_schema.sql` below), but it backs the unrelated Basics tab's official content, not `community.tsx`. `supabase/sql/accounts_progress_community.sql` describes an alternate anonymous-auth/profiles-based schema that was proposed early on but **not adopted** — the app uses the schema above instead. Keep that file as historical reference only, not as a plan to implement.
- `supabase/sql/v2_content_schema.sql` **is the live schema** for how language content is stored, replacing the per-language-table pattern (`letters_japanese`/`words_*`/`word_descriptions`) with a tree: `languages` (root) → `categories` (content_type — `letter`/`word`/`unique_feature` — lives directly on the category row, not a separate lookup table) → content. `letters` (kana) references a `letter_types` table (scoped per language, e.g. Japanese's hiragana/katakana) instead of hardcoding hiragana/katakana columns. All three content types — `letters`, `words`, and `kanji` — have a matching `*_translations` child table (rows per target language, not per-language columns, so adding a language later is an insert, not a schema change) rather than a hardcoded `romaji` column or translation columns. **Confirmed run against the live DB, and the app reads from it now** (`lib/cardCashe.ts`, `lib/categories.ts`, `lib/languages.ts`) — the old tables above are left in place, untouched, but no longer read by the app.
- `supabase/sql/v2_unique_features.sql` is a **follow-up to v2_content_schema.sql** (run after it, not instead of it) that reroutes kanji out of being its own top-level content_type and into a new `unique_features` registry — a plug-in point for language-specific things that don't cleanly fit the letter/word split (kanji is kind of a word, kind of a letter). `categories.content_type` becomes `letter`/`word`/`unique_feature`; a `unique_feature_id` column identifies *which* quirk (e.g. Japanese's `kanji` row) when content_type is `unique_feature`. `kanji` gets the same `unique_feature_id` denormalized directly on it. `kanji_readings`/`kanji_translations` are untouched by this file. Confirmed run against the live DB.
- `supabase/sql/v2_migrate_kana_from_live.sql` is the **real bulk content pass for kana**: reads (never writes) the live `letters_japanese` table and migrates every non-kanji row (`has_kanji = false`) into the v2 tree - one `categories` row per distinct old `category` value, up to two `letters` rows per old row (hiragana/katakana split via `letter_type_id`), and a `letter_translations` row per new letter carrying the old `romaji` as an English translation. Old "hiragana"/"katakana" tag values are stripped on the way in since `letter_type_id` already encodes that. Every insert has an `on conflict do nothing` matched to the real constraint, so it's safe to re-run and safe to run after `v2_seed_test_data.sql` (the 2 test letters just get skipped as already-present). Kanji migration is a separate pass (see below). Confirmed run against the live DB.
- `supabase/sql/v2_migrate_words_from_live.sql` is the **real bulk content pass for words**: reads (never writes) `words_japanese`/`words_english`/`word_descriptions` and migrates every row into the v2 tree - one `categories` row per distinct old `category`, a `words` row per old row, and a `word_translations` row carrying the English text plus the old `description` as `notes`. `words_chinese` is deliberately skipped (unused by app code, no `zh` language row exists yet). Assumed `word_descriptions.tags` existed (i.e. `add_tags_to_word_descriptions.sql` had been run) - would have errored clearly if not, and it didn't, so that column was confirmed present. Idempotent via a `NOT EXISTS` guard (words has no natural unique constraint like letters does). Confirmed run against the live DB.
- `supabase/sql/v2_migrate_kanji_from_live.sql` is the **real bulk content pass for kanji**, and unlike the kana/words migrations it's necessarily incomplete: reads (never writes) `letters_japanese` (`has_kanji = true` rows) and populates `kanji` + `kanji_readings`, but leaves `kanji_translations` **empty on purpose** - the old schema never captured any English meaning for kanji, only a single unlabeled reading (migrated in as `reading_type = 'other'`, not a real onyomi/kunyomi guess). Category is derived from the JLPT-level tag (e.g. `n5`) rather than the old `category` column, which was just the redundant literal string `'kanji'` on every row. Meanings were authored fresh as real content work (see `v2_seed_kanji_meanings.sql` next). Idempotent (NOT EXISTS / ON CONFLICT DO NOTHING). Confirmed run against the live DB.
- `supabase/sql/v2_seed_kanji_meanings.sql` fills the gap `v2_migrate_kanji_from_live.sql` deliberately left: English meanings (as `kanji_translations` rows) for all 109 JLPT N5 kanji. This is fresh content, not a migration - the old schema never had a meaning field for kanji at all. Sourced by cross-checking JLPT Sensei's N5 kanji reference for ~80 of the 109, with the rest (common basics like 安/新/多/少/立/買/会/言/英/社/店/足/耳/力/道/駅/魚/肉) given directly. Must run after `v2_migrate_kanji_from_live.sql` (matches kanji by character). Idempotent. Confirmed run against the live DB.
- `supabase/sql/v2_add_kanji_reading_romaji.sql` adds a `romaji` column to `kanji_readings` and backfills it for all 109 kanji - discovered missing only once the app integration needed it (the live "kanji → romaji" quiz mode in `offline.tsx`'s `LETTER_MODES` has nothing to read otherwise). Backfilled from the same recovered source as `v2_seed_kanji_meanings.sql`, not re-derived. Run after `v2_migrate_kanji_from_live.sql`. Confirmed run against the live DB.
- `supabase/sql/v2_diagnose_empty_results.sql` - read-only diagnostic (row counts + RLS status for every v2 table in one query). Useful any time the app appears to get zero results from a v2 table with no error - see the next entry for why that happens.
- `supabase/sql/v2_add_read_policies.sql` - **confirmed necessary and applied**: unlike `letters_japanese`/`words_*`/`word_descriptions` (no RLS at all), Supabase auto-enabled RLS on every v2 table with zero policies once they were created, which made every query silently return an empty result (not an error) for the app's anon key even though the migrated data was genuinely all there (confirmed via the diagnostic above: 167 words, 188 letters, 109 kanji, etc.). This file adds a blanket "anyone can read" SELECT policy to every v2 table (no write policies - same reasoning as the rest of this content: public reference data, written by hand via the SQL editor, not by the app). **If touching v2 tables again in the future and something is unexpectedly empty, check RLS status first** (`v2_diagnose_empty_results.sql`) before assuming a migration didn't run.
- `supabase/sql/v2_progress_schema.sql` replaces the old `user_progress` table's role (not the table itself, which is left alone) with `letter_progress`/`word_progress`/`kanji_progress` - one table per content type, each with a real FK into `letters`/`words`/`kanji` instead of a loose `item_id` nothing enforces. `accuracy` is a generated column (`correct_count / exposures`), not a manually-recomputed running average. `self_report` is dropped (dead per the notes above). Unlike the content tables, these have real per-user RLS (`auth.uid() = user_id`), not blanket public read. **Deliberately not migrated from the old `user_progress`** - starting fresh, since old ids don't resolve against the new content tables anyway. The app (`lib/progress.ts`, `lib/stats.ts`) now writes/reads these instead of `user_progress`. Confirmed run against the live DB.
- `lib/stats.ts`'s `fetchStatsSummary` now queries `word_progress`/`letter_progress`/`kanji_progress` directly (each already scoped to the right content type by which table it is), joining each row to its content table's `categories(key)` for the per-category breakdown, instead of the old approach of pulling every row from a single `user_progress` and cross-checking an opaque `item_id` against `word_descriptions`/`letters_japanese` to figure out what it even was. kanji rows are folded into `contentType: 'letter'`, category `'kanji'`, matching the pseudo-category the Flash Cards UI already treats it as. The exported `StatsSummary`/`CategoryStat` shapes are unchanged, so `app/(tabs)/stats.tsx` needed no changes.
- `supabase/sql/v2_lessons_schema.sql` adds `lessons` (scoped to a language, same as `categories`) and `lesson_sections` (the actual reading content, one row per section in `sort_order`, not one text blob per lesson - lets a lesson be edited/reordered section-by-section later). Both get the same "Anyone can read" RLS policy the rest of the v2 content tables needed (added directly in this file, not as a separate follow-up, now that the RLS-gotcha is known up front - see the `v2_add_read_policies.sql` entry above). `supabase/sql/v2_seed_lessons.sql` seeds the first two lessons (`hiragana-basics`, `katakana-basics`), original content written for this app. `lib/lessons.ts` (`fetchLessons`, `fetchLessonDetail`) and the `app/(tabs)/learn.tsx` rewrite (see below) read from these. Confirmed run against the live DB.
- `supabase/sql/v2_views.sql` creates `letters_with_translations`, a view joining `letters` to `letter_types`/`categories`/`languages`/`letter_translations` (kanji excluded - it has its own shape). Not hardcoded to any language pair - exposes `language_code` and `target_language_code` as plain columns so any (source, target) pair works by filtering the query, not editing the view. One row per (letter, translation); a letter with no translation yet still appears once with those columns null. Confirmed run against the live DB, but **not currently used by any app code** — `lib/cardCashe.ts` queries the base tables directly instead, since it needs to pair hiragana/katakana rows by shared romaji within a category, which this view doesn't do. Keep as an ad hoc query aid, not a load-bearing dependency.
- `supabase/sql/v2_seed_test_data.sql` inserts exactly 2 real rows into every v2 leaf/child table (letters, letter_translations, words, word_translations, kanji, kanji_readings, kanji_translations; `categories` gets 3 — one per content_type) as a smoke test of the whole tree before the real bulk content pass — あ/ア ("a"), 猫/犬 ("cat"/"dog"), 一/二 ("one"/"two"). Assumes v2_content_schema.sql, v2_unique_features.sql, and v2_schema_hardening.sql have all been run, and is deliberately shaped to satisfy the hardening pass's triggers/constraints. Not idempotent (meant to run once against empty tables) — has a commented-out cleanup block at the bottom to remove just this test data before the real pass. Confirmed run against the live DB — **the cleanup block was not confirmed as run**, so these 2 test rows per table (あ/ア, 猫/犬, 一/二) are likely still mixed into the real migrated content. Low-stakes (harmless, on-brand test data, not corruption), but worth knowing about if row counts ever look "off by a couple" or these specific characters show up somewhere unexpected.
- `supabase/sql/v2_schema_hardening.sql` is a **follow-up to v2_content_schema.sql** (independent of v2_unique_features.sql) closing gaps flagged in review: partial unique indexes so at most one row can be `is_primary` per (item, target language) on `word_translations`/`kanji_translations`/`letter_translations`/`kanji_readings`; composite indexes on those same tables tuned to the actual "translation of X into Y" query instead of two single-column indexes; triggers (`enforce_language_matches_category`, `enforce_letters_language_consistency`) that reject inserts/updates where a row's `language_id` doesn't match what its `category_id` (and for letters, `letter_type_id`) implies, since that denormalized `language_id` had nothing enforcing it stayed consistent; and a unique constraint on `letters (language_id, letter_type_id, character)` to stop duplicate glyphs. No structural changes, purely additive. Confirmed run against the live DB. Still open/undecided: whether an item should be able to belong to more than one structured category (currently one `category_id` + free-text `tags[]`) - that would need a many-to-many join table and wasn't part of this pass.
- `supabase/sql/v2_seed_more_animals.sql` adds 12 more words to the 'animals' category (鳥/魚/馬/牛/豚/羊/うさぎ/象/熊/猿/ライオン/パンダ), which previously had only the 2 smoke-test rows (猫/犬) from `v2_seed_test_data.sql` — the real bulk word migration never touched it since old live data had no "animals" category. Pure content authoring, idempotent (NOT EXISTS guard on text+category, matching the migration scripts' pattern). Not yet confirmed run against the live DB.
- `supabase/sql/v2_add_lesson_type.sql` adds a `lesson_type` column to `lessons` (`'fundamentals' | 'practical'`, default `'fundamentals'` — Postgres backfills existing rows via the default, no separate UPDATE needed). Distinguishes the original hand-authored reading lessons (hiragana-basics, katakana-basics) from the new "Practical Lessons" category (real-world scenario lessons, read one step at a time — see `app/(tabs)/learn.tsx` below). Confirmed run against the live DB.
- `supabase/sql/v2_add_lesson_blanks.sql` adds `lesson_section_blanks` — at most one fill-in-the-blank exercise per `lesson_sections` row (enforced by a `unique (lesson_section_id)` constraint), storing the sentence split around the blank (`prompt_before`/`prompt_after`), accepted answers in **both** kana and romaji (`answer_kana text[]`/`answer_romaji text[]` — either counts as correct, not one canonical script), and `decoy_words text[]` for the tap-to-fill word bank (display-only, never graded). Same "Anyone can read" RLS pattern as the rest of the v2 content tables. **Gotcha discovered here, worth remembering for any future to-one embed:** because `lesson_section_id` has a `unique` constraint, PostgREST embeds this table as a single object (or `null`), not an array, when queried as `lesson_sections.select('...,lesson_section_blanks(...)')` — the first version of `lib/lessons.ts`'s `fetchLessonDetail` assumed it was always an array (`(row.lesson_section_blanks ?? [])[0]`), which silently returned `undefined` with **no error at all**, so the blanks just never showed up client-side even though the data and RLS were both completely correct. Fixed via `normalizeBlankRow()`, which handles either shape. Confirmed run against the live DB.
- `supabase/sql/v2_seed_basic_phrases_lesson.sql` seeds the first real Practical Lesson — "Basic Phrases" (`lesson_type = 'practical'`), 9 steps: an intro, seven everyday greetings (hello/how are you/I'm fine/thank you/nice to meet you/what's your name/goodbye) each with a `lesson_section_blanks` row, and a review step. Idempotent (`NOT EXISTS` guards). Confirmed run against the live DB.
- `supabase/sql/v2_views_security_invoker.sql` fixes a Supabase security-advisor warning on `letters_with_translations` ("defined with the SECURITY DEFINER property") — `v2_views.sql`'s plain `create view` defaults to running as its creator (the SQL-editor's `postgres` role, which bypasses RLS) rather than the querying role. Sets `security_invoker = true` (Postgres 15+) so the view enforces the querying role's own RLS instead. Pure safety tightening, no behavior change (the view is unused by app code and the underlying tables already allow public read). **Not yet confirmed run against the live DB** — ask before assuming.
- `bug_reports` (id, created_at, title, description, email, page, user_id, status) — backs the "Report Bug" tab (`supabase/sql/bug_reports.sql`). RLS only has an **insert** policy for anon+authenticated — deliberately no select policy, so submissions are only readable by the developer via the Supabase dashboard (service role bypasses RLS). Confirmed run against the live DB (the table exists and the Report Bug tab works). Note the separate `bug_reports_require_login.sql` (mentioned in the report.tsx bullet below) tightens this insert policy to `authenticated`-only — confirm that one's been run too if `anon` submissions ever start appearing again.

- `supabase/sql/reconstruct_schema.sql` is a **consolidated, schema-only snapshot** of the current live database structure (no data, no seed rows) - the flattened end state of every v2_*.sql file plus the still-used v1 tables (`users`, `bug_reports`) and the still-present-but-superseded v1 tables (`letters_japanese`/`words_*`/`word_descriptions`/`user_progress`), assembled into one file for version control. Deliberately excludes `content_items`/`lesson_ratings`/`lesson_stats`/`copyright_claims`/`sessions` - these showed up in a live schema visualizer screenshot but match nothing in this repo or the rest of AGENTS.md (almost certainly leftovers from the original pre-v2 construct plan or the unadopted `accounts_progress_community.sql` design) - excluded per explicit user choice, not an oversight. The v2/progress/lessons sections are exact (copied from this repo's own files); the v1 section (F) has no original creation script anywhere in this repo's history (made directly via the Supabase table editor) so it's reconstructed from this doc's column lists, best-effort, not verified against a real `pg_dump`. Not meant to be run against the live DB as a routine migration - it's a from-scratch structural snapshot, safe to run only against an empty database.

All one-off SQL lives in `supabase/sql/*.sql` and is meant to be run manually in the Supabase SQL editor — nothing auto-runs. **As of 2026-07-06, every file that existed as of that date was confirmed run against the live DB** by the user (except `v2_seed_more_animals.sql` and `reconstruct_schema.sql`, added that same day). Four more files were added 2026-07-07 (`v2_add_lesson_type.sql`, `v2_add_lesson_blanks.sql`, `v2_seed_basic_phrases_lesson.sql` — all three confirmed run; `v2_views_security_invoker.sql` — not yet confirmed). For any *new* SQL file added after this point, still ask the user to confirm it's been run before assuming so, since there's no way to query the live DB from this environment (no network egress to Supabase from the sandbox) — see Environment quirks below.

## App progress (by screen)

- **`app/login.tsx`** — working Supabase email/password auth, handles the "signUp succeeds but returns no session because email confirmation is required" case (shows an info message instead of redirecting), supports `?mode=signup`, calls `ensureUserRow` after any successful session.
- **`app/(tabs)/offline.tsx` ("Flash Cards")** — the most developed screen. Three-column layout on desktop (left panel is word categories + letter categories, including kanji's "exclusive focus" toggle behavior; center is the active flashcard; right panel is a difficulty-cap stepper and tag filter). Below the `MOBILE_BREAKPOINT` (see responsive layout note below) it stacks into one scrolling column instead: card first, then categories, then difficulty/tags. Difficulty-cap stepper runs 1 up to the live max difficulty in the data; tag filter is free-text input + bubbles + random suggestions pulled from live data, validated on Apply — an unrecognized tag shows a "Tag: 'x' doesn't exist" error card instead of silently returning nothing. Filters persist across visits via `lib/studyPreferences.ts` (AsyncStorage). Session queue logic lives in `lib/cardQueue.ts`: cards get "rested" after 3 correct answers in a session (deprioritized, not discarded) and are recycled back into rotation if a filter combo's fresh pool runs out, so a small category (e.g. one kanji difficulty tier) can't hard-stop the session. Decoys are tracked by row id (`lib/cardQueue.ts`'s `DecoyEntry`) so a decoy can never come from the same row as the correct answer. Progress writes to `letter_progress`/`word_progress`/`kanji_progress` via `lib/progress.ts`'s `recordAnswer`, skipped entirely when the tap-to-hear hint was used that card (mute state is irrelevant to whether the hint "counts" as used). A toggle on the card switches between "Announce correct" (speaks the real reading after every correct answer, still counts normally) and "Announce Celebration" (random Japanese exclamation, or a user-picked custom sound file if enabled in Settings) — both respect mute; its labels shorten to "Correct"/"Celebrate" when narrow.
- **`components/flashcardcomponent.tsx`** — renders the actual card. Particle pronunciation overrides (は→"wa" / へ→"e" for particle-category word cards only, not letter-mode cards, which should still say "ha"/"he"), celebration phrases, and the TTS locale string passed to `Speech.speak()` now all come from `lib/languageConfig.ts`'s `getLanguageConfig(card.language)` rather than being hardcoded Japanese constants in this file — see the Language architecture entry below. Category label is hidden for letter cards (categories like "s-row" are internal, not meant to be a visible hint). `getSpokenQuestion` special-cases two situations where the literal question text shouldn't be spoken as-is: letter/kanji cards whose `questionScript` is `'kanji'` speak the row's `hiragana` reading instead of the bare glyph (a kanji character is genuinely ambiguous to TTS — most have several possible readings); and cards whose `questionScript` is `'romaji'` speak the kana reading instead of the raw romaji text. **The second case was the resolved cause of the "some kanji sound different" report from 2026-07-06** — `LETTER_MODES`/`languageConfig.letterModes` only restricts the `'kanji'` script to `hasKanji` rows, not `'romaji'`, so a kanji row could land on a `{question:'romaji', answer:'hiragana'}` mode; with no fallback, `Speech.speak()` was fed the raw romaji string (e.g. "nani") and read it as Latin letters with a Japanese voice instead of speaking the actual pronunciation — reproduced concretely as 何 (romaji "nani") sounding like "ka." Fixed 2026-07-07. Card sizing is responsive via `useIsNarrow()` — narrow width shrinks `mainText` from 72px to 44px and tightens the card's padding/minHeight, since the desktop sizing overflowed/crowded phone-width screens.
- **`app/(tabs)/stats.tsx`** — overall accuracy/exposures summary plus a per-category breakdown (every word category and every `LETTER_CATEGORIES` entry, defaulting to 0% if untouched rather than being omitted). Signed-out users get an empty state with a Sign Up prompt.
- **`app/(tabs)/index.tsx` ("Home")** — static hand-written Updates/Upcoming sections, no live data. Update this by hand whenever something user-visible ships or slips — it's not generated from anything.
- **`app/(tabs)/community.tsx`** — UI shell only, left-nav with Japanese Forum / English Forum / Lessons sub-views, no queries, "Create Lesson" button disabled with a note that Claude will eventually be wired in here to help users author/run custom lessons. **This is the biggest unbuilt feature** — nothing behind it is real yet. Left nav is responsive: on desktop it's a 200px side column, below `MOBILE_BREAKPOINT` it moves above the content and lays its three nav items out horizontally (wrapping) instead of a tall vertical list, and the "Community" panel title is hidden to save vertical space.
- **`app/(tabs)/learn.tsx` ("Basics")** — no longer a stub, and as of 2026-07-07 has two lesson categories via a left nav (same responsive collapse-to-horizontal-bar pattern as `community.tsx`'s left nav — see `lib/lessons.ts`'s `LessonType`):
  - **Fundamentals** — the original curated reading lessons (`hiragana-basics`, `katakana-basics` — textbook-standard ordering, script fundamentals before any grammar/vocab lesson exists), read as one long scrolling page (`FundamentalsDetail`), unchanged from before Practical Lessons existed. An "app orientation" lesson is still explicitly deferred since the app itself is actively changing.
  - **Practical Lessons** — real-world scenario lessons (first one: `basic-phrases`, see `v2_seed_basic_phrases_lesson.sql` above), read one `lesson_sections` row ("step") at a time (`PracticalLessonView`) instead of one long page. Steps are plain reading — page-turn arrows (‹ ›) anchored to the step card's own bottom corners move between them (not a full-width Back/Next row); the last step's forward arrow just disables rather than "finishing"/exiting, since there's no longer a concept of the lesson closing on its own. Leaving is only ever the "‹ Practical Lessons" link at the top.
  - Below the step card, **`LessonTestPanel`** is always rendered (not gated behind reaching the last step, and not hidden once you're on step 1) — an "open book" test gathering every step's fill-in-the-blank question (`lesson_section_blanks`, if any) into one panel on the same scrollable page, so the lesson content stays visible/scrollable-to while answering instead of a question disappearing once you page past its step. Each question renders via **`BlankExercise`**, which supports two interaction modes switched globally in Settings (`lib/preferences.tsx`'s `exerciseInputMode: 'tap' | 'type'`, defaulting to `'tap'`):
    - `'tap'` — fill the blank by tapping a shuffled word bubble (the correct answer plus `decoy_words`); wrong taps vanish from the bank, same pattern as `flashcardcomponent.tsx`'s multiple choice.
    - `'type'` — a free-text input, graded via `lib/lessonBlanks.ts`'s `isCorrectBlankAnswer` (accepts either kana or romaji, normalized/trimmed, case-insensitive for romaji — neither script is treated as more canonical, per an explicit decision since some learners won't have a Japanese IME set up).
  - Getting a question wrong never blocks anything — the whole Test panel is just content on the page, not a gate.
  - Both categories share the same `lessons`/`lesson_sections` tables and the same list-then-detail flow (`lib/lessons.ts`'s `fetchLessons`/`fetchLessonDetail`) — only the detail rendering differs by `lesson.lessonType`.
  - Distinct from the "community lessons" idea sitting inside `community.tsx` (user-authored/social, still fully deferred) — this is all official, hand-authored content.
  - **Known gap:** none of this writes to any progress table — Test panel answers are ephemeral (reset on leaving the lesson), unlike Flash Cards' `letter_progress`/`word_progress`/`kanji_progress`. Not addressed yet.
- **`app/(tabs)/settings.tsx`** — Appearance (light/dark/system), Celebration Sound (toggle + `expo-document-picker` file picker for a custom "Announce Celebration" sound, copied into permanent storage via `expo-file-system` on native; on web the picker's own data-URI is used directly since there's no real filesystem to copy into), Learning (reset flashcard filters), Account (Sign In / Sign Up when logged out, email + Sign Out when logged in — guest browsing is the default, nobody is logged in unless they choose to be).
- **`lib/preferences.tsx`** — global, AsyncStorage-persisted app prefs: theme mode, mute, announce mode (`'correct' | 'celebration'`), custom celebration sound enabled flag + uri + display name, and (as of 2026-07-07) `exerciseInputMode` (`'tap' | 'type'` — see the Practical Lessons entry above). None of this is in Supabase — it's device-local only. **Theme cold-start/resume bug, fixed 2026-07-07:** `scheme` used to derive straight from React Native's `useColorScheme()` hook when `themeMode === 'system'`. On the web build specifically (this app has no native build — see Known gaps), that hook's underlying `prefers-color-scheme` listener can miss the change event when the tab/device was fully backgrounded or asleep, and can be briefly stale right at cold hydration too — so the app kept rendering whatever scheme was current when it was last active, and stayed stuck that way until something forced a re-render with an explicit (non-`'system'`) `themeMode` (e.g. visiting Settings and picking Light/Dark, which bypasses system detection entirely and made it look like a fix when really nothing had been re-synced). Fix: `liveSystemScheme` state that explicitly re-reads `Appearance.getColorScheme()` on mount and on `visibilitychange`/`focus` (web-only), not just whatever the passive hook last reported.
- **Language architecture (`lib/languageConfig.ts` + `lib/activeLanguage.tsx`), added 2026-07-07** — scaffolding so a second language is "add a config object + seed its content in Supabase" rather than a grep-and-replace sweep, per an explicit request to prepare for this *without* actually adding a second language yet. `lib/languageConfig.ts` is a `LANGUAGE_CONFIGS` registry (only `'ja'` registered today) holding each language's TTS locale, celebration phrases, particle pronunciation overrides, valid quiz-mode script pairs (replacing `offline.tsx`'s old hardcoded `LETTER_MODES` constant), and its "unique feature" pseudo-category key (`'kanji'` for Japanese). `lib/activeLanguage.tsx` is an AsyncStorage-persisted context (same pattern as `preferences.tsx`) tracking which language is active, defaulting to Japanese — there's no UI to actually change it yet. `cardCashe.ts`, `categories.ts`, `offline.tsx`, `flashcardcomponent.tsx`, and `learn.tsx` all thread a `languageCode` param / read `card.language` through this instead of hardcoding `'ja'` inline. **Explicit scope boundary:** `LetterCard`'s field shape (hiragana/katakana/romaji/kanji as fixed fields — see `cards.ts`) and the `kanji`/`kanji_readings` tables themselves are still Japanese-specific — a language with a fundamentally different script system needs that data shape widened too, which is a separate, bigger change not attempted here.
- **`lib/lessonBlanks.ts`, added 2026-07-07** — grading (`isCorrectBlankAnswer`) and word-bank (`buildWordBank`, `displayAnswer`) helpers for Practical Lessons' fill-in-the-blank questions, shared between `BlankExercise`'s two input modes. See the Practical Lessons entry above.
- **App/web icons, added 2026-07-07** — `assets/icon.png` and `assets/favicon.png` regenerated as a red "L" on a black rounded-square background (previously a generic placeholder). `app/(tabs)/_layout.tsx`'s tab bar now sets a `tabBarIcon` per tab (was unset entirely, which showed a broken-image glyph on mobile web) — plain emoji glyphs (🏠📊📖💬🎴🐞⚙️), not an icon font/library, since `@expo/vector-icons` isn't installed and npm installs are flaky in the sandbox this gets edited in (see Environment quirks).
- **`app/(tabs)/report.tsx` ("Report Bug")** — free-text bug report form (title, description, optional email), writing to `bug_reports` via `lib/bugReports.ts`. **Requires login** — guests see a Sign In/Sign Up prompt instead of the form. This is enforced twice: the UI gate here, and `bug_reports`' RLS insert policy only grants to the `authenticated` role (`bug_reports_require_login.sql` tightened this after it was initially open to `anon` too — if `bug_reports.sql` hasn't been re-run since, make sure the require-login script has been). Submission is additionally gated by a client-side-only "human check" (a randomly generated small-number addition question, regenerated on a wrong answer) plus a hidden honeypot field — **not** a real server-verified captcha. That was an explicit tradeoff: a real captcha (reCAPTCHA/Turnstile) would need a Supabase Edge Function to verify the token server-side (the app's first real backend piece), which the user opted out of in favor of zero extra setup. Good enough to stop naive/scripted spam, not a determined attacker hitting the Supabase insert endpoint directly (though requiring login at least ties every report to an account now) — if spam becomes a real problem, revisit with a real captcha + Edge Function.

## Mobile-web responsive layout

The web build renders directly in phone browsers (no native app yet), so it needs to hold up at phone width, not just shrink a desktop-fixed layout. `lib/responsive.ts` exports `MOBILE_BREAKPOINT = 700` and a `useIsNarrow()` hook (`useWindowDimensions` based) — this is a plain width check, not a device/platform check, so a narrow desktop browser window gets the same treatment as an actual phone, which is the correct behavior here. Screens/components changed to use it:

- **`app/(tabs)/offline.tsx`** — 3-column desktop layout stacks into one scrolling column below the breakpoint (card, then categories, then difficulty/tags). See the offline.tsx bullet above for specifics.
- **`app/(tabs)/community.tsx`** — 200px fixed left nav becomes a full-width bar above the content, nav items go horizontal/wrapping instead of vertical.
- **`app/(tabs)/_layout.tsx`** — the 7-tab top tab bar was overflowing/cramping at phone width. Converted to a `TAB_CONFIG` array with `title`/`shortTitle` pairs (e.g. "Flash Cards"→"Cards", "Report Bug"→"Report", "Community"→"Comm."), rendered dynamically; narrow width also shrinks `tabBarLabelStyle` fontSize to 10 and tightens `tabBarItemStyle` padding.
- **`components/flashcardcomponent.tsx`** — `mainText` shrinks from 72px to 44px and the card's padding/minHeight tighten below the breakpoint, so a large kanji glyph or a long romaji string doesn't overflow/crowd a ~375px-wide screen.

Screens deliberately **left unchanged** after audit — `settings.tsx`, `stats.tsx`, `report.tsx`, `index.tsx`, `login.tsx` — because they already use a single-centered-column pattern (`maxWidth` + `width:'100%'` + `alignSelf:'center'`) that degrades gracefully without a breakpoint.

Deliberate choice: layout switches use plain conditional JSX (separate `...Narrow` style objects and, where needed, reordered elements), not CSS `order` — that's react-native-web-only and isn't reliably supported by React Native's Yoga layout engine, and native (iOS/Android) builds are an explicit next phase for this project, so the responsive approach needed to stay portable rather than being a web-only hack.

## Deployment

## Environment quirks worth knowing

- The sandbox this project is edited in has **no network egress to Supabase** — SQL/schema state can't be queried live from here; ask the user or rely on their screenshots/statements instead of trying to `curl` the Supabase REST API.
- The mounted project folder occasionally serves a stale/cached read to `mcp__workspace__bash` right after a `Write`/`Edit` call (seen on `package.json`, various `.tsx` files). Fix: `mv file file.bak && mv file.bak file` immediately before trusting a `cat`/`tsc` read of a just-edited file.
- Some directories on this mount (seen with `node_modules/expo/node_modules/expo-file-system` and a stray `dist/` folder) reject `npm`'s internal rename/rmdir with `EPERM`/`EACCES` even though direct file edits work fine. Workaround used successfully: copy the package contents to the top-level `node_modules/<pkg>` path manually and add the dependency to `package.json` by hand, rather than fighting `npm install`'s rename step.
- Always run `npx tsc --noEmit -p tsconfig.json` after edits (with the cache-refresh trick above) before calling something done.
- **`.git/objects/pack/multi-pack-index` was found corrupted on 2026-07-06** (`git status`/`git log` failed with `error: improper chunk offset(s)` / `fatal: unable to read <hash>`) — a newer pack file existed on disk (from an earlier commit) that the multi-pack-index hadn't been regenerated to include. Same `EPERM`/`EACCES` issue as the `node_modules` quirk above blocked deleting or overwriting it from the sandbox (`rm`, `mv`, and `git multi-pack-index write` all failed with "Operation not permitted"). **This was NOT fixed from the sandbox** — since the sandbox mounts the same real folder as the user's machine, this file may still be corrupted for their own local git client too, and could block their next `git add`/`commit`/`push`. If git commands fail with an "improper chunk offset" or similar low-level error, tell the user to delete `.git\objects\pack\multi-pack-index` themselves (from a normal terminal, not this sandbox) — git works fine without it and will regenerate it automatically later; this is a safe, non-destructive fix (it's just a lookup cache, not object data).
