# Project state (verified 2026-07-05 ‚Äî see lingrazul_construct.docx for the older, partly-superseded original plan)

This section is the source of truth for where the app actually is. The original construct plan (`lingrazul_construct.docx`) described a `content_items` / `user_progress` / `lessons` schema with a Node+Express backend ‚Äî that plan is largely historical at this point; don't assume anything from it without checking here first. Re-verify this section against the live Supabase project and codebase before relying on it for anything load-bearing ‚Äî it reflects a point-in-time check, not a live sync.

## Stack

No backend exists ‚Äî the Expo client (`lib/supabase.ts`) talks directly to Supabase using `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY` from `.env` (gitignored). No Vercel/Railway, no Whisper, no Claude API integration. Web is deployed via **EAS Hosting** (see Deployment section below) ‚Äî `expo.web.output` is set to `"static"` in `app.json`.

## Supabase schema (live)

- `letters_japanese` (id, hiragana, katakana, romaji, kanji, has_kanji, difficulty, category, tags text[]). Populated: base kana (vowel/k-row/.../w-row/n-standalone), dakuten/handakuten, youon (`insert_youon_letters.sql`), loanword-only extended katakana (`insert_katakana_extended_letters.sql`), and ~109 JLPT N5 kanji rows with one reading each (`insert_n5_kanji.sql`, category `kanji`, difficulty 4 ‚Äî schema only supports one reading per row, flagged as a known simplification).
- `words_japanese` (id, text), `words_english` (id, text), `words_chinese` (id, text ‚Äî unused by app code).
- `word_descriptions` (id, description, category, difficulty, tags text[] ‚Äî joins to the `words_*` tables by shared id). The `tags` column was added late via `supabase/sql/add_tags_to_word_descriptions.sql` (`alter table ... add column tags text[] default '{}'`) ‚Äî **check whether this script has actually been run against the live DB before assuming word cards have a tags column.** Even once run, no real tag content has been authored for word rows yet (they'll all be empty arrays) ‚Äî that's a separate content-curation pass, distinct from the schema change.
- `users` (id ‚Äî must equal the `auth.users` UID, email, style_preference, created_at). Auto-provisioned via a DB trigger (`users_auto_provision.sql`) plus a client-side self-healing fallback (`lib/users.ts`'s `ensureUserRow`, called from `login.tsx` after every successful sign-in). `style_preference` is for an onboarding flow that was never built ‚Äî safe to leave null.
- `user_progress` (user_id, item_id, exposures, accuracy, self_report, last_seen). `item_id` intentionally has **no foreign key** (`drop_user_progress_item_fk.sql` dropped it) because it can point at either `word_descriptions` or `letters_japanese` and there's no `content_type` column to disambiguate ‚Äî code checks both tables when needed (see `lib/stats.ts`). `accuracy` is a weighted 0‚Äì1 ratio, not a raw count. `self_report` is now dead weight ‚Äî the "Too hard"/"Too easy" buttons that wrote to it were removed from the UI (redundant once the difficulty filter existed), but the column and `recordAnswer`-adjacent write path were never dropped from the DB; nothing currently writes `self_report`.
- No `lessons`, `lesson_ratings`, `lesson_stats`, `copyright_claims`, or `sessions` tables ‚Äî Community is UI-only (see below). `supabase/sql/accounts_progress_community.sql` describes an alternate anonymous-auth/profiles-based schema that was proposed early on but **not adopted** ‚Äî the app uses the schema above instead. Keep that file as historical reference only, not as a plan to implement.
- `bug_reports` (id, created_at, title, description, email, page, user_id, status) ‚Äî backs the "Report Bug" tab (`supabase/sql/bug_reports.sql`). RLS only has an **insert** policy for anon+authenticated ‚Äî deliberately no select policy, so submissions are only readable by the developer via the Supabase dashboard (service role bypasses RLS). **Check whether this script has actually been run** before assuming the table exists.

All one-off SQL lives in `supabase/sql/*.sql` and is meant to be run manually in the Supabase SQL editor ‚Äî nothing auto-runs. Before starting new DB work, it's worth asking the user which of these have actually been run, since there's no way to query the live DB from this environment (no network egress to Supabase from the sandbox) ‚Äî see Environment quirks below.

## App progress (by screen)

- **`app/login.tsx`** ‚Äî working Supabase email/password auth, handles the "signUp succeeds but returns no session because email confirmation is required" case (shows an info message instead of redirecting), supports `?mode=signup`, calls `ensureUserRow` after any successful session.
- **`app/(tabs)/offline.tsx` ("Flash Cards")** ‚Äî the most developed screen. Three-column layout on desktop (left panel is word categories + letter categories, including kanji's "exclusive focus" toggle behavior; center is the active flashcard; right panel is a difficulty-cap stepper and tag filter). Below the `MOBILE_BREAKPOINT` (see responsive layout note below) it stacks into one scrolling column instead: card first, then categories, then difficulty/tags. Difficulty-cap stepper runs 1 up to the live max difficulty in the data; tag filter is free-text input + bubbles + random suggestions pulled from live data, validated on Apply ‚Äî an unrecognized tag shows a "Tag: 'x' doesn't exist" error card instead of silently returning nothing. Filters persist across visits via `lib/studyPreferences.ts` (AsyncStorage). Session queue logic lives in `lib/cardQueue.ts`: cards get "rested" after 3 correct answers in a session (deprioritized, not discarded) and are recycled back into rotation if a filter combo's fresh pool runs out, so a small category (e.g. one kanji difficulty tier) can't hard-stop the session. Decoys are tracked by row id (`lib/cardQueue.ts`'s `DecoyEntry`) so a decoy can never come from the same row as the correct answer. Progress writes to `user_progress` via `lib/progress.ts`'s `recordAnswer`, skipped entirely when the tap-to-hear hint was used that card (mute state is irrelevant to whether the hint "counts" as used). A toggle on the card switches between "Announce correct" (speaks the real reading after every correct answer, still counts normally) and "Announce Celebration" (random Japanese exclamation, or a user-picked custom sound file if enabled in Settings) ‚Äî both respect mute; its labels shorten to "Correct"/"Celebrate" when narrow.
- **`components/flashcardcomponent.tsx`** ‚Äî renders the actual card. Handles „ÅØ‚Üí"wa" / „Å∏‚Üí"e" pronunciation overrides for particle-category word cards only (not letter-mode cards, which should still say "ha"/"he"). Category label is hidden for letter cards (categories like "s-row" are internal, not meant to be a visible hint). `getSpokenQuestion` also special-cases letter cards whose `questionScript` is `'kanji'`: it speaks the row's `hiragana` reading instead of the bare kanji glyph, since a single kanji character is genuinely ambiguous to TTS engines (most have several possible readings) and asking `Speech.speak()` to read the glyph directly was a coin flip on whether it said the intended reading at all. **Open/unconfirmed:** the user reported afterwards that "some kanji" still sound different from the letter/kana answers ‚Äî this wasn't reproduced or investigated further (get concrete examples: which kanji, which mode, what was heard vs. expected) before ending that session. Could be a residual app bug, or could just be an installed-TTS-voice quality issue outside the app's control ‚Äî worth digging into with specifics next time it comes up. Card sizing is responsive via `useIsNarrow()` ‚Äî narrow width shrinks `mainText` from 72px to 44px and tightens the card's padding/minHeight, since the desktop sizing overflowed/crowded phone-width screens.
- **`app/(tabs)/stats.tsx`** ‚Äî overall accuracy/exposures summary plus a per-category breakdown (every word category and every `LETTER_CATEGORIES` entry, defaulting to 0% if untouched rather than being omitted). Signed-out users get an empty state with a Sign Up prompt.
- **`app/(tabs)/index.tsx` ("Home")** ‚Äî static hand-written Updates/Upcoming sections, no live data.
- **`app/(tabs)/community.tsx`** ‚Äî UI shell only, left-nav with Japanese Forum / English Forum / Lessons sub-views, no queries, "Create Lesson" button disabled with a note that Claude will eventually be wired in here to help users author/run custom lessons. **This is the biggest unbuilt feature** ‚Äî nothing behind it is real yet. Left nav is responsive: on desktop it's a 200px side column, below `MOBILE_BREAKPOINT` it moves above the content and lays its three nav items out horizontally (wrapping) instead of a tall vertical list, and the "Community" panel title is hidden to save vertical space.
- **`app/(tabs)/learn.tsx` ("Basics")** ‚Äî intentionally left alone as a stub per explicit earlier instruction; not a priority.
- **`app/(tabs)/settings.tsx`** ‚Äî Appearance (light/dark/system), Celebration Sound (toggle + `expo-document-picker` file picker for a custom "Announce Celebration" sound, copied into permanent storage via `expo-file-system` on native; on web the picker's own data-URI is used directly since there's no real filesystem to copy into), Learning (reset flashcard filters), Account (Sign In / Sign Up when logged out, email + Sign Out when logged in ‚Äî guest browsing is the default, nobody is logged in unless they choose to be).
- **`lib/preferences.tsx`** ‚Äî global, AsyncStorage-persisted app prefs: theme mode, mute, announce mode (`'correct' | 'celebration'`), custom celebration sound enabled flag + uri + display name. None of this is in Supabase ‚Äî it's device-local only.
- **`app/(tabs)/report.tsx` ("Report Bug")** ‚Äî free-text bug report form (title, description, optional email), writing to `bug_reports` via `lib/bugReports.ts`. **Requires login** ‚Äî guests see a Sign In/Sign Up prompt instead of the form. This is enforced twice: the UI gate here, and `bug_reports`' RLS insert policy only grants to the `authenticated` role (`bug_reports_require_login.sql` tightened this after it was initially open to `anon` too ‚Äî if `bug_reports.sql` hasn't been re-run since, make sure the require-login script has been). Submission is additionally gated by a client-side-only "human check" (a randomly generated small-number addition question, regenerated on a wrong answer) plus a hidden honeypot field ‚Äî **not** a real server-verified captcha. That was an explicit tradeoff: a real captcha (reCAPTCHA/Turnstile) would need a Supabase Edge Function to verify the token server-side (the app's first real backend piece), which the user opted out of in favor of zero extra setup. Good enough to stop naive/scripted spam, not a determined attacker hitting the Supabase insert endpoint directly (though requiring login at least ties every report to an account now) ‚Äî if spam becomes a real problem, revisit with a real captcha + Edge Function.

## Mobile-web responsive layout

The web build renders directly in phone browsers (no native app yet), so it needs to hold up at phone width, not just shrink a desktop-fixed layout. `lib/responsive.ts` exports `MOBILE_BREAKPOINT = 700` and a `useIsNarrow()` hook (`useWindowDimensions` based) ‚Äî this is a plain width check, not a device/platform check, so a narrow desktop browser window gets the same treatment as an actual phone, which is the correct behavior here. Screens/components changed to use it:

- **`app/(tabs)/offline.tsx`** ‚Äî 3-column desktop layout stacks into one scrolling column below the breakpoint (card, then categories, then difficulty/tags). See the offline.tsx bullet above for specifics.
- **`app/(tabs)/community.tsx`** ‚Äî 200px fixed left nav becomes a full-width bar above the content, nav items go horizontal/wrapping instead of vertical.
- **`app/(tabs)/_layout.tsx`** ‚Äî the 7-tab top tab bar was overflowing/cramping at phone width. Converted to a `TAB_CONFIG` array with `title`/`shortTitle` pairs (e.g. "Flash Cards"‚Üí"Cards", "Report Bug"‚Üí"Report", "Community"‚Üí"Comm."), rendered dynamically; narrow width also shrinks `tabBarLabelStyle` fontSize to 10 and tightens `tabBarItemStyle` padding.
- **`components/flashcardcomponent.tsx`** ‚Äî `mainText` shrinks from 72px to 44px and the card's padding/minHeight tighten below the breakpoint, so a large kanji glyph or a long romaji string doesn't overflow/crowd a ~375px-wide screen.

Screens deliberately **left unchanged** after audit ‚Äî `settings.tsx`, `stats.tsx`, `report.tsx`, `index.tsx`, `login.tsx` ‚Äî because they already use a single-centered-column pattern (`maxWidth` + `width:'100%'` + `alignSelf:'center'`) that degrades gracefully without a breakpoint.

Deliberate choice: layout switches use plain conditional JSX (separate `...Narrow` style objects and, where needed, reordered elements), not CSS `order` ‚Äî that's react-native-web-only and isn't reliably supported by React Native's Yoga layout engine, and native (iOS/Android) builds are an explicit next phase for this project, so the responsive approach needed to stay portable rather than being a web-only hack.

Deploy flow: `npx expo export --platform web` then `eas deploy` (preview) or `eas deploy --prod` (production, `https://lingrazul.expo.app`). ## Environment quirks worth knowing

- The sandbox this project is edited in has **no network egress to Supabase** ‚Äî SQL/schema state can't be queried live from here; ask the user or rely on their screenshots/statements instead of trying to `curl` the Supabase REST API.
- The mounted project folder occasionally serves a stale/cached read to `mcp__workspace__bash` right after a `Write`/`Edit` call (seen on `package.json`, various `.tsx` files). Fix: `mv file file.bak && mv file.bak file` immediately before trusting a `cat`/`tsc` read of a just-edited file.
- Some directories on this mount (seen with `node_modules/expo/node_modules/expo-file-system` and a stray `dist/` folder) reject `npm`'s internal rename/rmdir with `EPERM`/`EACCES` even though direct file edits work fine. Workaround used successfully: copy the package contents to the top-level `node_modules/<pkg>` path manually and add the dependency to `package.json` by hand, rather than fighting `npm install`'s rename step.
- Always run `npx tsc --noEmit -p tsconfig.json` after edits (with the cache-refresh trick above) before calling something done.
