# Project state (verified 2026-07-07 — see lingrazul_construct.docx for the older, partly-superseded original plan)

This section is the source of truth for where the app actually is. The original construct plan (`lingrazul_construct.docx`) described a `content_items` / `user_progress` / `lessons` schema with a Node+Express backend — that plan is largely historical at this point; don't assume anything from it without checking here first. Re-verify this section against the live Supabase project and codebase before relying on it for anything load-bearing — it reflects a point-in-time check, not a live sync.

## Stack

No backend exists — the Expo client (`lib/supabase.ts`) talks directly to Supabase using `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY` from `.env` (gitignored). No Whisper, no Claude API integration. Web is deployed via **Netlify** as of 2026-07-07, having moved off EAS Hosting (see Deployment section below) — `expo.web.output` is set to `"static"` in `app.json`.

## Supabase schema (live)

- `letters_japanese` (id, hiragana, katakana, romaji, kanji, has_kanji, difficulty, category, tags text[]). Populated: base kana (vowel/k-row/.../w-row/n-standalone), dakuten/handakuten, youon (`insert_youon_letters.sql`), loanword-only extended katakana (`insert_katakana_extended_letters.sql`), and ~109 JLPT N5 kanji rows with one reading each (`insert_n5_kanji.sql`, category `kanji`, difficulty 4 — schema only supports one reading per row, flagged as a known simplification).
- `words_japanese` (id, text), `words_english` (id, text), `words_chinese` (id, text — unused by app code).
- `word_descriptions` (id, description, category, difficulty, tags text[] — joins to the `words_*` tables by shared id). The `tags` column was added late via `supabase/sql/add_tags_to_word_descriptions.sql` (`alter table ... add column tags text[] default '{}'`) — confirmed run against the live DB. No real tag content has been authored for word rows though (they're all empty arrays) — that's a separate content-curation pass, distinct from the schema change, and still outstanding.
- `users` (id — must equal the `auth.users` UID, email, style_preference, created_at). Auto-provisioned via a DB trigger (`users_auto_provision.sql`) plus a client-side self-healing fallback (`lib/users.ts`'s `ensureUserRow`, called from `login.tsx` after every successful sign-in). `style_preference` is for an onboarding flow that was never built — safe to leave null.
- `user_progress` (user_id, item_id, exposures, accuracy, self_report, last_seen). `item_id` intentionally has **no foreign key** (`drop_user_progress_item_fk.sql` dropped it) because it can point at either `word_descriptions` or `letters_japanese` and there's no `content_type` column to disambiguate. **Superseded as of `v2_progress_schema.sql`** — the app no longer reads or writes this table at all (`lib/progress.ts`/`lib/stats.ts` use `letter_progress`/`word_progress`/`kanji_progress` instead, see below). Left in place, untouched, not migrated from. `self_report` was already dead weight before the supersession — the "Too hard"/"Too easy" buttons that wrote to it were removed from the UI (redundant once the difficulty filter existed), and the column was dropped entirely in the v2 replacement.
- No `lesson_ratings`, `lesson_stats`, `copyright_claims`, or `sessions` tables — `community.tsx` (messaging/forums/user-authored lessons) is UI-only (see below). A `lessons`/`lesson_sections` pair DOES now exist (see `v2_lessons_schema.sql` below), but it backs the unrelated Basics tab's official content, not `community.tsx`. `supabase/sql/accounts_progress_community.sql` describes an alternate anonymous-auth/profiles-based schema that was proposed early on but **not adopted** — the app uses the schema above instead. Keep that file as historical reference only, not as a plan to implement.
- `supabase/sql/v2_content_schema.sql` **is the live schema** for how language content is stored, replacing the per-language-table pattern (`letters_japanese`/`words_*`/`word_descriptions`) with a tree: `languages` (root) → `categories` (content_type — `letter`/`word`/`unique_feature` — lives directly on the category row, not a separate lookup table) → content. `letters` (kana) references a `letter_types` table (scoped per language, e.g. Japanese's hiragana/katakana) instead of hardcoding hiragana/katakana columns. All three content types — `letters`, `words`, and `kanji` — have a matching `*_translations` child table (rows per target language, not per-language columns, so adding a language later is an insert, not a schema change) rather than a hardcoded `romaji` column or translation columns. **Confirmed run against the live DB, and the app reads from it now** (`lib/cardCashe.ts`, `lib/categories.ts`, `lib/languages.ts`) — the old tables above are left in place, untouched, but no longer read by the app.
- `supabase/sql/v2_unique_features.sql` is a **follow-up to v2_content_schema.sql** (run after it, not instead of it) that reroutes kanji out of being its own top-level content_type and into a new `unique_features` registry — a plug-in point for language-specific things that don't cleanly fit the letter/word split (kanji is kind of a word, kind of a letter). `categories.content_type` becomes `letter`/`word`/`unique_feature`; a `unique_feature_id` column identifies *which* quirk (e.g. Japanese's `kanji` row) when content_type is `unique_feature`. `kanji` gets the same `unique_feature_id` denormalized directly on it. `kanji_readings`/`kanji_translations` are untouched by this file. Confirmed run against the live DB.
- `supabase/sql/v2_migrate_kana_from_live.sql` is the **real bulk content pass for kana**: reads (never writes) the live `letters_japanese` table and migrates every non-kanji row (`has_kanji = false`) into the v2 tree - one `categories` row per distinct old `category` value, up to two `letters` rows per old row (hiragana/katakana split via `letter_type_id`), and a `letter_translations` row per new letter carrying the old `romaji` as an English translation. Old "hiragana"/"katakana" tag values are stripped on the way in since `letter_type_id` already encodes that. Every insert has an `on conflict do nothing` matched to the real constraint, so it's safe to re-run and safe to run after `v2_seed_test_data.sql` (the 2 test letters just get skipped as already-present). Kanji migration is a separate pass (see below). Confirmed run against the live DB.
- `supabase/sql/v2_migrate_words_from_live.sql` is the **real bulk content pass for words**: reads (never writes) `words_japanese`/`words_english`/`word_descriptions` and migrates every row into the v2 tree - one `categories` row per distinct old `category`, a `words` row per old row, and a `word_translations` row carrying the English text plus the old `description` as `notes`. `words_chinese` is deliberately skipped (unused by app code, no `zh` language row exists yet). Assumed `word_descriptions.tags` existed (i.e. `add_tags_to_word_descriptions.sql` had been run) - would have errored clearly if not, and it didn't, so that column was confirmed present. Idempotent via a `NOT EXISTS` guard (words has no natural unique constraint like letters does). Confirmed run against the live DB.
- `supabase/sql/v2_migrate_kanji_from_live.sql` is the **real bulk content pass for kanji**, and unlike the kana/words migrations it's necessarily incomplete: reads (never writes) `letters_japanese` (`has_kanji = true` rows) and populates `kanji` + `kanji_readings`, but leaves `kanji_translations` **empty on purpose** - the old schema never captured any English meaning for kanji, only a single unlabeled reading (migrated in as `reading_type = 'other'`, not a real onyomi/kunyomi guess). Category is derived from the JLPT-level tag (e.g. `n5`) rather than the old `category` column, which was just the redundant literal string `'kanji'` on every row. Meanings were authored fresh as real content work (see `v2_seed_kanji_meanings.sql` next). Idempotent (NOT EXISTS / ON CONFLICT DO NOTHING). Confirmed run against the live DB.
- `supabase/sql/v2_seed_kanji_meanings.sql` fills the gap `v2_migrate_kanji_from_live.sql` deliberately left: English meanings (as `kanji_translations` rows) for all 109 JLPT N5 kanji. This is fresh content, not a migration - the old schema never had a meaning field for kanji at all. Sourced by cross-checking JLPT Sensei's N5 kanji reference for ~80 of the 109, with the rest (common basics like 安/新/多/少/立/買/会/言/英/社/店/足/耳/力/道/駅/魚/肉) given directly. Must run after `v2_migrate_kanji_from_live.sql` (matches kanji by character). Idempotent. Confirmed run against the live DB.
- `supabase/sql/v2_add_kanji_reading_romaji.sql` adds a `romaji` column to `kanji_readings` and backfills it for all 109 kanji - discovered missing only once the app integration needed it (the live "kanji → romaji" quiz mode in `offline.tsx`'s `LETTER_MODES` has nothing to read otherwise). Backfilled from the same recovered source as `v2_seed_kanji_meanings.sql`, not re-derived. Run after `v2_migrate_kanji_from_live.sql`. Confirmed run against the live DB.
- `supabase/sql/v2_diagnose_empty_results.sql` - read-only diagnostic (row counts + RLS status for every v2 table in one query). Useful any time the app appears to get zero results from a v2 table with no error - see the next entry for why that happens.
- `supabase/sql/v2_add_read_policies.sql` - **confirmed necessary and applied**: unlike `letters_japanese`/`words_*`/`word_descriptions` (no RLS at all), Supabase auto-enabled RLS on every v2 table with zero policies once they were created, which made every query silently return an empty result (not an error) for the app's anon key even though the migrated data was genuinely all there (confirmed via the diagnostic above: 167 words, 188 letters, 109 kanji, etc.). This file adds a blanket "anyone can read" SELECT policy to every v2 table (no write policies - same reasoning as the rest of this content: public reference data, written by hand via the SQL editor, not by the app). **If touching v2 tables again in the future and something is unexpectedly empty, check RLS status first** (`v2_diagnose_empty_results.sql`) before assuming a migration didn't run.
- `supabase/sql/v2_progress_schema.sql` replaces the old `user_progress` table's role (not the table itself, which is left alone) with `letter_progress`/`word_progress`/`kanji_progress` - one table per content type, each with a real FK into `letters`/`words`/`kanji` instead of a loose `item_id` nothing enforces. `accuracy` is a generated column (`correct_count / exposures`), not a manually-recomputed running average. `self_report` is dropped (dead per the notes above). Unlike the content tables, these have real per-user RLS (`auth.uid() = user_id`), not blanket public read. **Deliberately not migrated from the old `user_progress`** - starting fresh, since old ids don't resolve against the new content tables anyway. The app (`lib/progress.ts`, `lib/stats.ts`) now writes/reads these instead of `user_progress`. Confirmed run against the live DB.

Not hardcoded to any language pair - exposes `language_code` and `target_language_code` as plain columns so any (source, target) pair works by filtering the query, not editing the view. ## App progress (by screen)

- **`app/login.tsx`** — working Supabase email/password auth, handles the "signUp succeeds but returns no session because email confirmation is required" case (shows an info message instead of redirecting), supports `?mode=signup`, calls `ensureUserRow` after any successful session.
- **`app/(tabs)/offline.tsx` ("Flash Cards")** — the most developed screen. Three-column layout on desktop (left panel is word categories + letter categories; center is the active flashcard; right panel is a difficulty-cap stepper and tag filter). Below the `MOBILE_BREAKPOINT` (see responsive layout note below) it stacks into one scrolling column instead: card first, then categories, then difficulty/tags. Fix: the initializer now always starts at `'light'` unconditionally (matching the static export's own fallback exactly, regardless of `rnSystemScheme`), and the existing effects below correct it to the real device scheme immediately after mount - which happens after hydration completes, so it's an ordinary client-side update rather than something compared against server markup. Trade-off: dark-mode visitors get one imperceptible light-then-dark flash on load instead of a console error.

**Theme cold-start/resume bug, fixed 2026-07-07:** `scheme` used to derive straight from React Native's `useColorScheme()` hook when `themeMode === 'system'`. On the web build specifically (this app has no native build — see Known gaps), that hook's underlying `prefers-color-scheme` listener can miss the change event when the tab/device was fully backgrounded or asleep, and can be briefly stale right at cold hydration too — so the app kept rendering whatever scheme was current when it was last active, and stayed stuck that way until something forced a re-render with an explicit (non-`'system'`) `themeMode` (e.g. visiting Settings and picking Light/Dark, which bypasses system detection entirely and made it look like a fix when really nothing had been re-synced). ## Mobile-web responsive layout

The web build renders directly in phone browsers (no native app yet), so it needs to hold up at phone width, not just shrink a desktop-fixed layout. `lib/responsive.ts` exports `MOBILE_BREAKPOINT = 700` and a `useIsNarrow()` hook (`useWindowDimensions` based) — this is a plain width check, not a device/platform check, so a narrow desktop browser window gets the same treatment as an actual phone, which is the correct behavior here. Screens/components changed to use it:

- **`app/(tabs)/offline.tsx`** — 3-column desktop layout stacks into one scrolling column below the breakpoint (card, then categories, then difficulty/tags). See the offline.tsx bullet above for specifics.
- **`app/(tabs)/community.tsx`** — 200px fixed left nav becomes a full-width bar above the content, nav items go horizontal/wrapping instead of vertical.
- **`app/(tabs)/_layout.tsx`** — the 7-tab top tab bar was overflowing/cramping at phone width. Converted to a `TAB_CONFIG` array with `title`/`shortTitle` pairs (e.g. "Flash Cards"→"Cards", "Report Bug"→"Report", "Community"→"Comm."), rendered dynamically; narrow width also shrinks `tabBarLabelStyle` fontSize to 10 and tightens `tabBarItemStyle` padding.
- **`components/flashcardcomponent.tsx`** — `mainText` shrinks from 72px to 44px and the card's padding/minHeight tighten below the breakpoint, so a large kanji glyph or a long romaji string doesn't overflow/crowd a ~375px-wide screen.

Screens deliberately **left unchanged** after audit — `settings.tsx`, `stats.tsx`, `report.tsx`, `index.tsx`, `login.tsx` — because they already use a single-centered-column pattern (`maxWidth` + `width:'100%'` + `alignSelf:'center'`) that degrades gracefully without a breakpoint.

Deliberate choice: layout switches use plain conditional JSX (separate `...Narrow` style objects and, where needed, reordered elements), not CSS `order` — that's react-native-web-only and isn't reliably supported by React Native's Yoga layout engine, and native (iOS/Android) builds are an explicit next phase for this project, so the responsive approach needed to stay portable rather than being a web-only hack.

## Environment quirks worth knowing

- The sandbox this project is edited in has **no network egress to Supabase** — SQL/schema state can't be queried live from here; ask the user or rely on their screenshots/statements instead of trying to `curl` the Supabase REST API.
- The mounted project folder occasionally serves a stale/cached read to `mcp__workspace__bash` right after a `Write`/`Edit` call (seen on `package.json`, various `.tsx` files). Fix: `mv file file.bak && mv file.bak file` immediately before trusting a `cat`/`tsc` read of a just-edited file.
- Some directories on this mount (seen with `node_modules/expo/node_modules/expo-file-system` and a stray `dist/` folder) reject `npm`'s internal rename/rmdir with `EPERM`/`EACCES` even though direct file edits work fine. Workaround used successfully: copy the package contents to the top-level `node_modules/<pkg>` path manually and add the dependency to `package.json` by hand, rather than fighting `npm install`'s rename step.
- Always run `npx tsc --noEmit -p tsconfig.json` after edits (with the cache-refresh trick above) before calling something done.
- **`.git/objects/pack/multi-pack-index` was found corrupted on 2026-07-06** (`git status`/`git log` failed with `error: improper chunk offset(s)` / `fatal: unable to read <hash>`) — a newer pack file existed on disk (from an earlier commit) that the multi-pack-index hadn't been regenerated to include. Same `EPERM`/`EACCES` issue as the `node_modules` quirk above blocked deleting or overwriting it from the sandbox (`rm`, `mv`, and `git multi-pack-index write` all failed with "Operation not permitted"). **This was NOT fixed from the sandbox** — since the sandbox mounts the same real folder as the user's machine, this file may still be corrupted for their own local git client too, and could block their next `git add`/`commit`/`push`. If git commands fail with an "improper chunk offset" or similar low-level error, tell the user to delete `.git\objects\pack\multi-pack-index` themselves (from a normal terminal, not this sandbox) — git works fine without it and will regenerate it automatically later; this is a safe, non-destructive fix (it's just a lookup cache, not object data).
